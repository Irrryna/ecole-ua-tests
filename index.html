🔄 Pattern CustomWorld

Chaque scénario dispose d'un monde isolé avec son propre contexte navigateur, ses données seedées et son état éphémère.

// Données partagées entre steps class CustomWorld { browser, context, page seedData ← créé avant chaque scénario invitedUser ← état éphémère invitation inviteePage ← page séparée (invité) newClassName, updatedTitle... }
Isolation totale — chaque scénario crée et nettoie ses propres données via l'API backend.

🐋 Docker — Mailpit

Intercepte tous les emails envoyés par le backend en mode test. Interface web pour inspecter les emails reçus.

# Lancer Mailpit docker run -d \ -p 1025:1025 \ # SMTP ← backend -p 8025:8025 \ # API HTTP ← tests --name mailpit \ axllent/mailpit
Pourquoi Mailpit ? Le backend utilise normalement Brevo (service email réel). En mode test, on reroute vers Mailpit pour intercepter les tokens d'activation sans envoyer de vrais emails.

🗄️ Base de données de test isolée

Base dédiée aux tests — séparée de la production. Données créées et nettoyées automatiquement à chaque run.

# backend/.env.test DATABASE_URL="postgresql://admin:*** @localhost:5432/ecole_test_db" MAIL_MODE="mailpit" MAILPIT_SMTP_HOST="localhost" MAILPIT_SMTP_PORT="1025" JWT_SECRET="test-jwt-secret-local-only"
Isolation totale — aucune donnée de production n'est touchée. Le hook AfterAll supprime toutes les données créées pendant les tests.

Scénario complexe — Invitation multi-acteurs avec email réel
Le test le plus avancé : deux contextes navigateur + interception d'email + token sécurisé
Étape 1 · Admin
Envoi de l'invitation
L'admin remplit le formulaire (prénom, nom, email, rôle) et clique "Envoyer l'invitation". Le backend crée un token UUID et envoie l'email via Mailpit SMTP.
Étape 2 · Email
Interception Mailpit
Le test interroge l'API Mailpit (GET /api/v1/messages), extrait le lien /reset-password?mode=invite&token=xxx du HTML de l'email.
Étape 3 · Invité
Activation du compte
Un second contexte navigateur (isolé de l'admin) ouvre le lien d'activation, crée un mot de passe et active le compte.
Étape 4 · Invité
Connexion et vérification dashboard
L'invité se connecte et le test vérifie qu'il atterrit sur le bon dashboard selon son rôle (Espace professeur / Espace parent).

🎯 Ce que ce test valide

✓ L'API d'invitation backend crée bien un token unique en base
✓ L'email HTML contient le bon lien avec le token
✓ La page d'activation accepte le token et active le compte
✓ La connexion post-activation fonctionne avec les nouveaux credentials
✓ Le rôle est correctement attribué (enseignant ≠ parent)
✓ La ligne d'invitation passe à "Invitation envoyée" dans le tableau admin
✓ L'isolement des sessions navigateur (admin et invité simultanément)
Pattern multi-acteurs — deux instances BrowserContext Playwright distinctes tournent en parallèle dans le même test.

🔑 Spécificités techniques résolues

Deux dépôts — Le dépôt de tests est public, l'application est privée. Le secret MAIN_REPO_TOKEN permet le clone du dépôt privé en CI.
Build NestJS — nest build génère dist/src/main.js (et non dist/main.js) à cause d'un fichier TypeScript à la racine.
Angular dev config — Le build utilise --configuration development pour pointer l'API sur localhost:3000 et non la production.
wait-on — Les tests démarrent uniquement après que backend et frontend sont accessibles, via npx wait-on.

📋 Extrait du workflow

# .github/workflows/browser-tests.yml services: postgres: image: postgres:15 mailpit: image: axllent/mailpit ports: ["1025:1025", "8025:8025"] steps: - name: Clone app (private repo) run: | git clone https://$MAIN_REPO_TOKEN@github.com/ Irrryna/ecole-ukrainienne-lyon.git - name: Start backend run: node dist/src/main & - name: Start frontend run: npx serve -s dist/frontend/browser & - name: Wait for services run: npx wait-on http://localhost:3000 http://localhost:4200 - name: Run Playwright tests run: npx playwright test
